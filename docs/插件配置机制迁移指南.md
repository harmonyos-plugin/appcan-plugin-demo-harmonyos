# AppCan鸿蒙版插件配置机制迁移指南

## 概述

- **时间**：2025年9月10日
- **引擎版本**：0.9.28

AppCan鸿蒙版引擎的插件配置机制已从旧的`plugin.json`格式升级到新的`plugin.config.ts`格式。本指南将帮助开发者了解变更内容并完成迁移。
原先需要在`appcan_extend_plugins.json`注册的操作可以省略了。

## 主要变更

### 1. 配置文件格式升级

**旧格式 (`plugin.json`)**:
```json
{
  "plugins": [
    {
      "uexName": "uexDemo",
      "methods": [
        "testFirstInterface",
        "openMainPage",
        "testHandleFileProtocolPath"
      ]
    }
  ]
}
```

**新格式 (`plugin.config.ts`)**:
```typescript
export const pluginConfig = {
  "plugins": [
    {
      "uexName": "uexDemo",
      "methods": [
        "testFirstInterface",
        "openMainPage",
        "testHandleFileProtocolPath"
      ]
    }
  ]
};
```

### 2. 插件模块导出更新

**`Index.ets` 文件需要添加配置导出**:
```typescript
import EUExDemo from './src/main/ets/EUExDemo';
import { pluginConfig } from './uexDemo/plugin.config.ts';

export {
  EUExDemo,
  pluginConfig
};
```

### 3. 中央注册器更新

**`AppCanExPluginProvider.ets` 需要导入和加载配置**:
```typescript
import { EUExDemo, pluginConfig as uexDemoConfig } from '@appcan/uexdemo';

export default class AppCanExPluginProvider implements AppCanPluginProvider {
  obtainPluginConfigs(context: Context): object[] {
    let plugin_configs = new Array<object>();
    plugin_configs = plugin_configs.concat(uexDemoConfig.plugins);
    return plugin_configs;
  }

  obtainInitializedPlugins(context: Context, brwView: EBrowserView): Map<string, EUExBase> {
    const pluginsClassMap: Map<string, EUExBase> = new Map();
    pluginsClassMap.set("uexDemo", new EUExDemo(context, brwView));
    return pluginsClassMap;
  }
}
```

## 迁移步骤

### 步骤1：创建新配置文件

1. 在插件目录下创建`plugin.config.ts`文件
2. 将`plugin.json`的内容转换为TypeScript格式
3. 确保导出`pluginConfig`常量

### 步骤2：更新模块导出

1. 修改插件的`Index.ets`文件
2. 添加`pluginConfig`的导入和导出

### 步骤3：更新中央注册器

1. 在`AppCanExPluginProvider.ets`中导入插件配置
2. 在`obtainPluginConfigs`方法中添加配置加载
3. 确保`obtainInitializedPlugins`中有对应的插件实例化

### 步骤4：验证和测试

1. 编译项目确保没有语法错误
2. 运行应用验证插件功能正常
3. 检查插件方法调用是否正确

## 注意事项

### 1. 向后兼容性
- 新机制不会影响现有插件的API接口
- JS端调用方式保持不变
- 插件功能行为保持一致

### 2. 配置验证
- 确保`uexName`与插件目录名一致
- 验证`methods`数组中的方法在入口类中都有实现
- 检查导入路径的正确性

### 3. 清理工作
- 可以选择保留或删除旧的`plugin.json`文件
- 更新相关文档中的配置文件引用
- 确保团队成员了解新的配置机制

## 常见问题

### Q: 是否需要立即删除`plugin.json`文件？
A: 不是必须的。新机制不依赖旧文件，可以选择保留作为备份或逐步清理。

### Q: 如何验证迁移是否成功？
A: 编译项目无错误，且插件功能在运行时正常工作即表示迁移成功。

### Q: 新机制是否影响插件的打包和分发？
A: 不影响。插件的打包和分发流程保持不变，只是配置文件格式发生了变化。

### Q: 如何处理第三方插件的迁移？
A: 第三方插件开发者需要按照本指南更新其插件配置，或者可以使用转换工具自动生成新格式配置。
